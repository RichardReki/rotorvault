# RotorVault — Plan 4: Web Dashboard — Implementation Plan

**Goal:** A polished, self-contained dashboard that shows (1) the **product** — live NAV, FXRP allocation
across venues, the FTSOv2 XRP/USD price + on-chain regime state, and the FDC-attested APY — and (2) the
**proof** — the backtest headline (risk overlay cuts XRP drawdown −78%→−44%) with an equity chart vs
baselines. **DEMO mode** works with no deployment (real committed backtest data + illustrative live
state); **LIVE mode** reads the deployed Coston2 contracts via viem once addresses are set.

**Architecture:** A single self-contained `web/index.html` (inline CSS/JS, no build step) — trivially
hostable (open the file, or Vercel/Pages). Real backtest equity/metrics are embedded (downsampled from
`backtest/results/`); LIVE mode dynamically imports viem from a CDN and reads RegimeGate/ApyOracle/Vault.

**Tech:** Vanilla HTML/CSS/JS + inline SVG charts + viem (ESM CDN, LIVE only). Dark theme, accessible
categorical colors (no rainbow), one clear visual hierarchy.

## Tasks
1. Extract a small real equity dataset (RotorVault OOS vs XRP HODL) + the metrics table from
   `backtest/results/dashboard_data.json`; embed as JSON constants.
2. `web/index.html` — layout: header + DEMO/LIVE toggle; Product panel (NAV, allocation bars, regime
   badge, FTSO price, APY); Proof panel (metrics table + SVG equity chart); Flare-integration footer.
3. LIVE mode: viem `coston2` client reads `RegimeGate.riskOn()/currentPrice1e18()`, `ApyOracle.apy()`,
   `RotorVault.totalAssets()`, venue `positionValue()`; addresses from a small config block (filled
   post-deploy). Graceful fallback to DEMO if reads fail.
4. `web/README.md` — how to view/host + the DEMO-vs-LIVE model.

## Notes
- Real numbers only: the metrics table and equity are the audited backtest results; live figures are
  labelled LIVE. No fabricated performance.
- Deploy addresses (Plan 2 Task 9) fill the LIVE config; until then DEMO is the default.
